## 1. Резюме

Портал «Цифровой двойник» — единая внутренняя платформа для дочерних и зависимых организаций (ДЗО) КМГ, объединяющая базу знаний, пресс-релизы, обучающие материалы, контакты участников рабочих групп и операционную канбан-доску по направлению Upstream. Цель — заменить разрозненные каналы (почта, локальные диски, чаты) единой точкой доступа к документации и координации по проекту внедрения цифровых двойников производственных активов.

На данный момент существует кликабельный HTML-прототип, реализующий целевой интерфейс и основные пользовательские сценарии на демо-данных. Документ описывает требования к продукту как для оценки прототипа, так и для перехода к промышленной реализации.

---

## 2. Проблема и контекст

Восемь ДЗО КМГ параллельно внедряют элементы цифрового двойника (моделирование активов, работа с телеметрией, прогнозная аналитика), но:

- регламенты и стандарты существуют в разрозненных версиях без единого источника правды;
- пресс-релизы и новости о ходе проекта не агрегируются, участники узнают о них с задержкой;
- обучающие материалы рассылаются вручную, нет единого каталога;
- нет актуального справочника контактов рабочих групп по каждому ДЗО;
- задачи по внедрению (на примере блока Upstream) ведутся вне единой системы, сложно отследить ответственных и статус.

## 3. Цели продукта

| Цель | Метрика успеха |
|---|---|
| Единая база знаний вместо разрозненных документов | 100% регламентов уровня группы КМГ размещены на портале в течение 1 квартала после запуска |
| Своевременное информирование о ходе проекта | Пресс-релиз публикуется на портале не позднее 1 рабочего дня после выпуска |
| Снижение времени поиска ответственного по ДЗО | Среднее время поиска контакта < 30 секунд (текущее — по оценке, несколько минут) |
| Прозрачность выполнения задач Upstream | 100% активных задач блока Upstream отражены на канбан-доске с указанием ответственного |
| Контролируемый доступ | Доступ к порталу только по корпоративной учётной записи |

### Вне целей (out of scope) на первом этапе

- Портал не заменяет промышленные SCADA/PLM-системы и не хранит сами модели цифровых двойников — только документацию, коммуникацию и координацию задач вокруг них.
- Канбан-доска на первом этапе покрывает только блок Upstream; распространение на другие блоки (Downstream, Midstream и т. д.) — предмет отдельного этапа.

## 4. Пользователи и роли

| Роль | Кто это | Ключевые сценарии |
|---|---|---|
| Администратор платформы | ИТ/куратор проекта цифрового двойника группы КМГ | Управление контентом всех разделов, наблюдение за всеми ДЗО |
| Руководитель рабочей группы ДЗО | Ответственный за внедрение в своём ДЗО | Публикация документов и задач своего ДЗО, контроль статуса задач, поиск контактов смежных ДЗО |
| Специалист рабочей группы | Инженер, аналитик, специалист по данным | Чтение базы знаний, прохождение обучения, работа с задачами, назначенными на него |
| Читатель (просмотр) | Смежные подразделения, руководство | Просмотр материалов без права редактирования *(роль предусмотрена, детальные права — см. открытые вопросы)* |

Для входа используется корпоративная учётная запись; в прототипе — демонстрационный набор логинов/паролей на стороне клиента (см. раздел 9.2).

## 5. Функциональные требования по разделам

### 5.1 Аутентификация

- Экран входа по логину и паролю перед доступом к любому разделу портала.
- Отображение сообщения об ошибке при неверных данных без указания, что именно неверно (логин или пароль).
- После входа — карточка пользователя (имя, роль) в боковой панели и возможность выйти из системы.
- **Прототип:** проверка логина/пароля происходит в браузере на фиксированном демо-списке пользователей — это заглушка без реальной защиты.
- **Требование к промышленной версии:** аутентификация должна выполняться на сервере через корпоративный SSO/Active Directory КМГ; пароли не должны существовать в клиентском коде ни в каком виде.

### 5.2 Главная страница

- Иллюстрация цифрового двойника: сопоставление физического объекта и цифровой модели (условная схема добывающего/перерабатывающего актива с датчиками и показаниями).
- Сводная статистика: количество подключённых ДЗО, документов базы знаний, обучающих материалов, участников рабочих групп.
- Панель связи ДЗО с платформой — визуальная схема подключения восьми ДЗО к единому контуру.
- Превью последних документов базы знаний (3) и последних пресс-релизов (3) с переходом в соответствующий раздел.

### 5.3 База знаний

- Список документов с полями: название, категория, ДЗО, дата публикации, краткое описание.
- Категории: технологии, стандарты, ИТ-инфраструктура, отраслевые практики, безопасность данных *(список расширяемый)*.
- Фильтрация по категории (чипы) и полнотекстовый поиск по названию/описанию/ДЗО.
- Просмотр документа в модальном окне с идентификатором записи (`KB-###`).
- **Открытый вопрос:** прототип не хранит сами файлы документов (PDF/DOCX) — только карточки-описания. Требуется решить, где физически хранятся файлы (например, интеграция с корпоративным хранилищем документов) и как будет работать загрузка/версионирование.

### 5.4 Пресс-релизы

- Лента новостей по проекту с полями: название, ДЗО, дата, краткое содержание.
- Фильтр по ДЗО, участие в общем поиске.
- Просмотр в модальном окне с идентификатором (`PR-###`).
- **Открытый вопрос:** нужен ли отдельный процесс модерации/согласования перед публикацией релиза (кто может публиковать, требуется ли утверждение пресс-службы КМГ)?

### 5.5 Обучающие материалы

- Каталог материалов с типами: курс, практикум, вебинар, пособие; поля — длительность, уровень сложности, описание.
- Фильтр по типу материала, участие в общем поиске.
- Просмотр карточки материала с идентификатором (`TR-###`).
- **Открытый вопрос:** нужен ли трекинг прохождения (кто прошёл какой курс, оценки/тесты, сертификация)? В прототипе не реализовано.

### 5.6 Контакты рабочих групп

- Список из восьми ДЗО, для каждого — сворачиваемая группа участников.
- Поля участника: ФИО, роль, e-mail, телефон.
- Участие в общем поиске (по имени, роли, названию ДЗО); при активном поиске группы разворачиваются автоматически.
- **Открытый вопрос:** источник данных — ручное ведение на портале или синхронизация с корпоративным HR-справочником/AD? Второе предпочтительно для промышленной версии, чтобы контакты не расходились с реальностью.

### 5.7 Канбан-доска Upstream

- Доска с колонками: «К выполнению», «В работе», «На проверке», «Выполнено».
- Карточка задачи: название, ДЗО, ответственный (с аватаром-инициалами), приоритет (высокий/средний/низкий), срок; просроченные задачи визуально выделяются.
- Перетаскивание карточек между колонками (drag-and-drop).
- Фильтр по ДЗО, участие в общем поиске.
- Добавление новой задачи через форму: название, ДЗО (Озенмунайгаз/Эмбамунайгаз), ответственный (список подтягивается по выбранному ДЗО и КМГ Инжиниринг), приоритет, срок; новая задача попадает в колонку «К выполнению».
- **Открытые вопросы:**
  - Нужны ли уведомления при назначении задачи или смене статуса?
  - Нужна ли доска только для Upstream или требуется тиражирование на другие производственные блоки?
  - Нужны ли комментарии/история изменений внутри карточки?

### 5.8 Общий поиск

- Единое поле поиска в верхней панели, работающее в контексте открытого раздела (документы, релизы, обучение, контакты, задачи).
- **Открытый вопрос:** нужен ли глобальный поиск сразу по всем разделам с единой выдачей, а не только в пределах текущего раздела?

## 6. Нефункциональные требования

| Категория | Требование |
|---|---|
| Доступ | Только для авторизованных сотрудников группы КМГ |
| Языки | Русский (интерфейс реализован на русском); при необходимости — казахский/английский, требует локализации |
| Устройства | Адаптивная вёрстка для десктопа и планшета; мобильная версия — упрощённая раскладка колонок |
| Производительность | Отклик интерфейса на действия пользователя (фильтрация, поиск) — не более 300 мс на клиенте |
| Доступность | Видимый фокус для клавиатурной навигации, учёт `prefers-reduced-motion` для анимаций |
| Хранение данных | В прототипе данные хранятся в памяти браузера и сбрасываются при перезагрузке страницы. Промышленная версия требует серверного хранилища данных (БД) и API |
| Безопасность | Серверная аутентификация (SSO/AD), разграничение прав по ролям и ДЗО, защищённая передача данных (HTTPS), журналирование действий пользователей |

## 7. Модель данных (укрупнённо)

```
ДЗО (id, название, отраслевой сегмент)

Документ базы знаний (id, название, категория, ДЗО, дата, описание, [файл])
Пресс-релиз        (id, название, ДЗО, дата, описание)
Обучающий материал (id, название, тип, длительность, уровень, описание)
Контакт             (id, ФИО, роль, ДЗО, e-mail, телефон)
Задача канбана       (id, название, ДЗО, ответственный, приоритет, срок, колонка)
Пользователь         (id, логин, роль, ДЗО, [хэш пароля — только на сервере])
```

Промышленная версия потребует отдельной проработки схемы БД, включая связи "многие-ко-многим" (например, один документ может относиться к нескольким ДЗО) и версионирование документов.

## 8. Пользовательские сценарии (примеры)

1. **Руководитель РГ Озенмунайгаза** заходит на портал, открывает канбан-доску Upstream, добавляет новую задачу «Калибровка модели пласта», назначает инженера своего ДЗО ответственным, выставляет высокий приоритет и срок.
2. **Специалист КМГ Инжиниринга** ищет через общий поиск «регламент обновления моделей», находит документ в базе знаний и открывает карточку для ознакомления.
3. **Новый участник рабочей группы** проходит вводный курс «Основы цифровых двойников в нефтегазовой отрасли» из раздела «Обучение».
4. **Сотрудник смежного ДЗО** находит контакт ответственного за цифровой двойник в Атырауском НПЗ через раздел «Рабочие группы».

## 9. Текущее состояние прототипа

### 9.1 Что уже реализовано

- Полностью кликабельный интерфейс всех разделов (5.1–5.8) на демонстрационных данных.
- Единая визуальная система (тёмная индустриальная тема, шрифты IBM Plex Mono/Sans).
- Drag-and-drop канбан-доска и модальные окна просмотра карточек.
- Экран входа с проверкой логина/пароля на стороне клиента.

### 9.2 Известные ограничения прототипа (техдолг перед промышленным запуском)

| Ограничение | Требуемое решение |
|---|---|
| Аутентификация — клиентская заглушка, пароли видны в исходном коде | Серверная аутентификация через SSO/AD КМГ |
| Данные не сохраняются между сессиями (только в памяти браузера) | Backend + база данных, REST/GraphQL API |
| Нет разграничения прав на редактирование (кто может публиковать документы/релизы) | Ролевая модель доступа (RBAC) на сервере |
| Нет хранения и загрузки файлов документов | Интеграция с корпоративным файловым хранилищем |
| Нет уведомлений (почта/мессенджер) о новых задачах и релизах | Сервис уведомлений, интеграция с корпоративной почтой/Teams |
| Демо-контакты и документы — иллюстративные | Наполнение реальными данными ДЗО перед запуском |

## 10. Метрики после запуска

- Доля активных пользователей от общего числа участников рабочих групп (DAU/WAU относительно списочного состава).
- Среднее время выполнения задачи на канбан-доске Upstream (от «К выполнению» до «Выполнено»).
- Количество документов базы знаний, обновлённых за квартал.
- Доля просроченных задач на доске.

## 11. Риски

| Риск | Влияние | Митигация |
|---|---|---|
| Разные ДЗО ведут документацию в разных форматах/стандартах | Несогласованность базы знаний | Единый регламент публикации, шаблоны документов |
| Низкая вовлечённость рабочих групп в поддержание актуальности контактов и задач | Устаревшие данные снижают доверие к порталу | Назначить ответственного координатора в каждом ДЗО, периодический аудит |
| Отсутствие интеграции с реальными системами телеметрии/SCADA на первом этапе | Портал остаётся справочно-координационным, не операционным инструментом | Явно зафиксировать это в scope, спланировать интеграцию отдельным этапом |
