# PRD: Пульт управления компанией для CEO

## 1\. Проблема

CEO получает данные о состоянии компании через еженедельные отчёты от департаментов, которые затем вручную консолидируются в сводки. Это создаёт задержку между тем, что происходит в компании, и тем, когда CEO об этом узнаёт, а также требует ручного труда на сборку отчётов каждую неделю.

## 2\. Цель

Единая точка входа, где CEO в реальном времени видит консолидированное состояние всех департаментов компании — без необходимости ждать сборки еженедельного отчёта. Департаменты вносят свои показатели напрямую, дашборд обновляется мгновенно.

## 3\. Пользователи и роли

|Роль|Что делает|Доступ|
|-|-|-|
|Сотрудник департамента|Вносит показатели своего департамента за период|Только форма ввода, не видит дашборд CEO|
|CEO|Просматривает консолидированный дашборд, видит риски и алармы|Только дашборд, без формы ввода|

Роли разделены на уровне интерфейса намеренно: это два разных инструмента с разными задачами, не два режима одного окна.

## 4\. Состав MVP

### 4.1 Департаменты (5 шт.)

Продажи, Маркетинг, Финансы, Операции, HR.

### 4.2 Метрики по департаментам

**Продажи**

* Выручка за период (факт), ₸
* План выручки, ₸
* Pipeline (открытые сделки), ₸
* Конверсия воронки, %
* Новые клиенты, шт

**Маркетинг**

* Новые лиды, шт
* CAC текущий, ₸
* CAC на прошлой неделе (для сравнения), ₸
* Освоено бюджета, ₸

**Финансы**

* Cash flow за период, ₸
* Маржинальность факт, %
* Маржинальность план, %
* Дебиторская задолженность, ₸
* Кредиторская задолженность, ₸

**Операции**

* Выполнение операционного плана, %
* Уровень брака, %
* Загрузка мощностей, %

**HR**

* Штат текущий, чел
* Открытые вакансии, шт
* Текучесть кадров, %
* Закрыто вакансий от плана, %

### 4.3 Продукт 1 — форма ввода (department-entry.html)

* Выбор департамента чипами
* Форма с полями метрик этого департамента, с текущими сохранёнными значениями
* Кнопка сохранения — данные сразу попадают в общее хранилище
* Нет доступа к консолидированному виду CEO

### 4.4 Продукт 2 — дашборд CEO (ceo-dashboard.html)

* **Сквозные KPI компании** (верхний ряд): выручка, маржинальность, cash flow, текучесть кадров — каждая с отклонением от плана/нормы
* **Блок «Риски и алармы»**:

  * Сводка: количество критичных / требующих внимания / департаментов без замечаний
  * Список алармов с указанием департамента, сути риска и конкретных цифр, подтверждающих его
  * Один департамент может иметь несколько одновременных алармов
* **Карточки департаментов**: статус-индикатор (зелёный/жёлтый/красный), ключевые метрики, время последнего обновления
* Кнопка ручного обновления данных
* Индикатор времени последнего обновления по компании в целом

## 5\. Логика рисков и алармов

Правила порогов зашиты в единой функции оценки рисков, чтобы их было легко калибровать.

|Департамент|Условие|Внимание (жёлтый)|Критично (красный)|
|-|-|-|-|
|Продажи|Факт/план выручки|80–95%|<80%|
|Продажи|Конверсия воронки|<10%|—|
|Маркетинг|Рост CAC к прошлой неделе|5–15%|>15%|
|Финансы|Маржа минус план|-1…-4 п.п.|хуже -4 п.п.|
|Финансы|Cash flow|—|отрицательный|
|Финансы|Дебиторка/кредиторка|соотношение >2|—|
|Операции|Выполнение плана|85–95%|<85%|
|Операции|Уровень брака|5–10%|>10%|
|Операции|Загрузка мощностей|>95%|—|
|HR|Текучесть кадров|5–10%|>10%|
|HR|Выполнение плана найма|<50% при открытых вакансиях|—|

Статус карточки департамента = наихудший из его алармов (если есть критичный — карточка красная, иначе если есть «внимание» — жёлтая, иначе зелёная).

**Примечание:** все пороги — рабочая гипотеза для MVP, требуют калибровки под реальные нормативы компании до внедрения в постоянную эксплуатацию.

## 6\. Технические ограничения MVP (важно)

Текущая реализация — прототип на key-value хранилище (window.storage), общем для обоих HTML-файлов:

* Нет аутентификации/авторизации по ролям — доступ к форме ввода не ограничен конкретным департаментом технически, полагается на то, что ссылка передаётся нужному человеку
* Нет истории изменений — хранится только последнее внесённое значение по каждому департаменту, нет графика динамики по неделям
* Нет журнала аудита (кто и когда изменил данные)
* Данные не синхронизированы с реальными источниками (CRM, 1С, банк) — вносятся вручную, что убирает часть ручного труда (сборку сводки), но не убирает ручной ввод цифр
* Не оптимизировано под множественных одновременных пользователей с конфликтами записи (last write wins)

## 7\. Что дальше (после MVP)

Приоритет для следующих итераций:

1. **Разграничение доступа** — логин по департаментам, чтобы ввод был технически ограничен своим разделом
2. **История и тренды** — хранение значений по периодам, графики план/факт в динамике на дашборде CEO
3. **Калибровка порогов алармов** — совместно с руководителями департаментов зафиксировать реальные нормативы
4. **Автоматизация сбора данных** — интеграция с системами, где данные уже существуют (CRM, бухгалтерия), чтобы сократить ручной ввод там, где это возможно
5. **Уведомления** — push/email алармов CEO при переходе метрики в критичную зону, не только по факту захода в дашборд

## 8\. Метрика успеха MVP

* CEO прекращает запрашивать еженедельные ручные сводки и использует дашборд как основной источник статуса компании
* Время от изменения показателя департаментом до появления на дашборде CEO — близко к нулю (вместо недели ожидания сводки)
* Все 5 департаментов регулярно (минимум раз в неделю) обновляют свои данные через форму
